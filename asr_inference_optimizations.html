01

Inference Server & Architecture

The server hosts a hybrid speech recognition model built around a single shared encoder and three independent decoders. The encoder — a deep stack of attention and convolution layers that converts raw audio into rich feature representations — is the dominant GPU cost. All three decoders consume the same encoder output and can be switched per-request with no model reload.

The encoder runs through a GPU-compiled execution plan that fuses operations and selects the fastest compute kernels automatically. The server's dynamic batcher groups requests arriving within a 20 ms window into a single shared encoder pass — spreading the expensive GPU compute across multiple simultaneous callers.

GPU Memory & Batch Safety

A single oversized batch can exhaust GPU memory and crash every subsequent request. Production systems defend at multiple levels: duration caps, pre-allocated workspace guards, and a health heartbeat decoupled from the inference path.

  • Audio-Duration Cap

    Before a batch reaches the GPU, total audio duration is summed across items. If it exceeds the threshold, the batch is split into sequential sub-batches — bounding peak activation memory regardless of how many long recordings arrive simultaneously.

  • Workspace Guard

    The step-by-step decoders reserve a fixed block of GPU memory during warm-up and lock their execution plan at that size. A hard cap of 8 items per sub-batch ensures the server never tries to use more memory than was reserved — preventing crashes from unexpected batch size spikes.

  • Health Heartbeat

    After every request the server writes a lightweight status record (GPU memory usage, call count, timestamp). A separate monitoring process reads this to report server health — without actually sending test audio — keeping health checks off the critical path.

  • Dynamic Batch Tuning

    The server batches up to 16 requests per GPU pass, preferring groups of 2, 4, 8, or 16. Reducing the maximum from 32 to 16 (part of our optimization pass) cut worst-case latency under burst traffic — smaller maximums prevent one large group from monopolizing the GPU.

Long-Audio Chunking & Stitching

Audio beyond ~20 s strains GPU memory and degrades the model's attention quality as the sequence grows longer. Rather than truncating or rejecting long recordings, audio is split at silence boundaries, decoded in overlapping chunks, and stitched together using word-level timestamps.

Waveform chunking visualization
Chunk 1 Chunk 2 Chunk 3 Overlap Silence

Split points are placed at silence centres. Each chunk overlaps its predecessor by ~2 s for acoustic context. Words are attributed to chunks by absolute timestamp.

  • 1
    Silence-aware split point

    Short-time energy < threshold for ≥ 150 ms. Split placed at the silence centre within a search window near the end of the chunk — never mid-word or mid-phoneme.

  • 2
    Overlap for acoustic context

    Each chunk starts ~2 s before the previous split point. The encoder needs preceding audio context to correctly decode words near the boundary. Overlapping decoded tokens are discarded during merge.

  • 3
    Timestamp-guided stitching

    Each chunk emits word-level timestamps. Words whose start time ≥ join point are attributed to the next chunk. Final transcript assembled chronologically from non-overlapping word sets.

Measured on 10-minute audio: Fast decoder completes in ~1.85 s. Balanced decoder in ~4.2 s (down from 7.3 s after optimization). Accurate decoder in ~10.9 s. All three scale linearly with audio duration.

Blank-Token Penalty

At each audio frame the decoder scores every token in the vocabulary including a special blank token meaning "no speech here." When the model over-predicts silence — due to constrained vocabulary or unfamiliar acoustics — short words get silently dropped. The penalty reduces the blank score, pushing the model toward emitting real tokens.

How blank penalty works — three variants
FLAT  — penalize every frame equally For each time frame in the audio:     Reduce the blank token's score by the penalty amount → Simplest form. Use when you want consistent blank suppression across the whole utterance.
CONFIDENCE-GATED  — penalize only when blank is strongly dominant For each time frame:     Compute gap = blank score − highest word-token score     Only if gap exceeds the threshold:         Reduce blank score by penalty amount → Selective. The model must already be very committed to silence before the penalty fires.
TAIL  — penalize only the final N frames For each frame in the last N frames of the audio (e.g. last 15 frames ≈ 600 ms):     Reduce blank score by penalty amount All earlier frames → left completely unchanged → Built for acoustic fusion. Targets the user mic segment only, leaving TTS audio unaffected.
  • Flat Uniform Penalty

    The blank score is reduced by a fixed amount at every audio frame. Cost is under 0.1 ms. Risk: if set too high (>5) it distorts genuinely silent frames. Start at 2.0–4.0 when using language constraining.

  • Conf Confidence-Gated

    Only fires when the gap between the blank score and the best word-token score exceeds a threshold — meaning the model is already very committed to silence. Adds ~0.5 ms. Safer for clean audio.

  • Tail Acoustic Fusion

    Applied only to the final N audio frames. Designed for voice-assistant scenarios where assistant speech is followed by user speech — penalizes exactly the low-confidence user region without touching the clean assistant segment.

Acoustic Fusion: Why the Tail Matters

In voice-assistant scenarios the server receives a single combined audio stream: the assistant's spoken response followed immediately by the user's reply. The speech recognition model was trained on clean studio-quality recordings — the assistant's synthesized speech matches this closely. The user's microphone audio, however, arrives with background noise and varied speaking style, so the model struggles to recognize it and defaults to "silence" at the end of the clip.

Language-Constrained Decoding

The model's internal word list — its vocabulary — is shared across all 9 supported languages, containing words and word-fragments from every script. Each entry is tagged with its language. When a caller specifies which languages are allowed, all entries from other languages are instantly blocked, turning the multilingual model into a focused single-language recognizer for that request.

Vocabulary mask — select languages to see which tokens survive
Showing full vocabulary (all languages active)
Tokens shown dimmed are fully blocked — they can never appear in the output. The language mask is computed once at startup and applied in a single fast operation before each decode step.
  • Instant Block (Fast decoder)

    The language filter is applied once in a single step before any decoding begins. Cost: 2–5 ms. Completely eliminates out-of-language words from the output — guaranteed.

  • Rescore After Decode (Default)

    Decode at full speed first with no language restriction, then go back and replace any wrong-language words using scores from the fast decoder at the same time position. Lower overhead for longer speech (≥1 s), but can miss errors on very short or ambiguous clips.

  • Enforce at Every Step (Strict)

    The language filter is applied inside the decoder at every single word-generation step, so wrong-language words are never even considered. More accurate for short clips or similar-sounding language pairs. The main optimization target — batching these strict-mode requests cut their latency by 47–49%.

Why bilingual mode is sometimes faster than single-language: A very narrow language filter leaves the decoder with very few word options at each step. When none of those options fit the audio well, the decoder stalls and takes more steps to recover. Allowing a second language widens the candidate pool, giving the decoder more valid options and letting it move forward efficiently.

Keyword Boosting & Context Biasing

Domain-specific terms — product names, medical terms, proper nouns — are rare in training data, so the model assigns them lower confidence than everyday words even when they're spoken clearly. Context biasing corrects this by adding a bonus score whenever the decoder's output matches a path through a keyword lookup structure, nudging it toward the correct specialized term.

Press Animate to start step-by-step prefix tree traversal.
How the prefix tree is built: Keywords are broken into smaller word-pieces using the same internal vocabulary as the speech model — this happens server-side, no local setup needed. Each keyword becomes a path through a prefix tree where each node represents a partial match. Silence frames act as self-loop edges, allowing the search to span silence within a keyword. The tree is built once per keyword list and reused across all requests.
  • Beam Traversal

    At each audio frame, active candidate paths advance along tree edges that match the current top-K token scores. When a path reaches a terminal node, the keyword is recorded with its accumulated score and time range. Low-scoring paths are pruned continuously to keep the search efficient.

  • Entropy-Weighted Boost

    The per-step bonus is scaled by the frame's uncertainty. More boost on ambiguous frames where the model is unsure — less on confident frames where it's already making the right choice. Separates the detection threshold from the per-step continuation bonus.

  • False-Positive Filter

    Detected keywords compete against the model's natural output for the same time region. If the model scores its own output higher than the keyword match plus a threshold, the keyword detection is discarded — preventing incorrect substitutions.

  • CPU / GPU Overlap

    In the optimized pipeline, the keyword candidate search (~20 ms of CPU work) is dispatched to a background worker thread just before the GPU decoder starts. Both run in parallel. Net residual cost: only the fast merge step (~1–2 ms) after both complete.

Word-Level Language Boosting

Blocking an entire language for a whole sentence is too blunt when people mix languages mid-sentence — a common pattern in multilingual India. Instead, this technique works word-by-word: for each recognized word, it detects which language the audio most resembles at that moment and boosts the scores of words from that language, without touching the rest of the sentence.

Mechanically: the decoder's word scores are divided into time windows, one per recognized word. For each window, scores are summed per language using a pre-built language annotation table. The dominant language is whichever scores highest in that window. Words from that language get a boost; words from other languages get a small penalty. The silence entry is always excluded from penalization.

Per-word language detection and boosting — algorithm
Repeat for every word in the decoded transcript:
Step 1  — detect the word's dominant language Look at all token scores within this word's time frames Sum the scores separately for each supported language Dominant language ← the language with the highest total score
Step 2  — boost the dominant language For every frame that belongs to this word:     Increase all tokens of the dominant language by lang_boost amount
Step 3  — penalize competing languages For every frame that belongs to this word:     Decrease all other-language token scores by lang_penalty amount     The blank token is always excluded from penalization
Example: "hospital" inside a Hindi sentence — English tokens get boosted in that word's frames only. The surrounding Hindi words are processed independently and left untouched.
In the optimized pipeline: the language-boosted decode step (~10 ms of CPU work) is dispatched to a background thread just before the GPU decoder starts — running in parallel with the ~30–40 ms GPU decode time. Its CPU cost is fully hidden.

Silent-Frame Pruning

A typical spoken utterance is 40–70% silence at the audio frame level. The step-by-step decoders must visit every frame in sequence — so feeding them silence wastes time. Removing high-confidence silent frames before decoding shortens the sequence they need to process, cutting latency directly in proportion to how much silence was pruned.

  • 1
    Blank-probability screening

    The blank token's probability is computed at every frame. Frames where it exceeds the threshold are flagged as silence candidates.

  • 2
    Boundary and representative preservation

    First and last valid frames always kept. Within long silent runs, at least one evenly-spaced representative frame is retained for positional continuity.

  • 3
    Index map for timestamp recovery

    A lookup table mapping each remaining frame back to its original position in the full audio is carried alongside the pruned sequence. When a word is recognized at frame k in the pruned sequence, the original timestamp is recovered via this map — no time information is lost.

Word-Level Timestamps & Confidence

Word-level timestamps unlock subtitle synchronization, speaker alignment, and per-word quality filtering. No extra model pass is needed — the decoder already records which audio frame each word was emitted at as a byproduct of its normal operation.

Word timestamp timeline visualization

Each block = one word. Width = duration. Colour = per-word confidence (green=high, amber=medium, red=low). Hover over words to see start/end times and scores.

  • Fast Decoder Timestamps

    After decoding, duplicates are merged and silence frames removed. Each surviving word-piece maps directly to a time offset (~40 ms per audio frame). Word-pieces are grouped into full words using vocabulary boundary markers. Zero added latency — timestamps are a free byproduct of the decode step.

  • Step-by-Step Decoder Timestamps

    During the word-by-word decode loop, the audio frame index at each word emission is recorded. These frame positions are grouped into words. If silent frames were pruned earlier, the original position lookup table is used to recover the correct absolute timestamp.

  • Per-Word Confidence Score

    The model's confidence for each recognized word-piece is averaged into a per-word score. Longer words contribute more to the overall utterance score — giving a stable quality signal that can be used to flag low-confidence words for review or filtering.

Python-Level Micro-Optimizations

The keyword matching search explores millions of partial candidate paths through the prefix tree per request. When this runs across thousands of audio frames and concurrent requests, even tiny per-step inefficiencies compound into measurable latency. These optimizations target the innermost loop.

Compact candidate-path objects Specialized boost closures Faster beam pruning without array allocation Batched vocabulary lookups Math value caching Thread-pool CPU/GPU overlap
  • Compact Candidate-Path Objects

    Hypothesis objects number in the millions per request. Declaring all attributes up front so the runtime stores them in fixed-size slots — instead of a general-purpose dictionary — removes ~200 bytes of overhead per object and speeds up allocation by ~30%, directly cutting memory pressure in the inner loop.

  • Pre-Compiled Scoring Functions

    The scoring bonus function is selected once at request init and bound as a mode-specialized callable. Eliminates 40+ mode-comparison checks per call — a compounding saving across millions of invocations per request.

  • Faster Beam Pruning

    Beam pruning previously converted each frame's scores into a temporary array just to find the maximum. Switching to a direct scan over the existing list avoids that allocation entirely. Across thousands of frames per request, these micro-savings compound into meaningful latency reduction.

  • Batched Vocabulary Lookups

    All unique word IDs in the sequence are collected first, then resolved in a single batch call. This reduces the number of lookups from one-per-frame to one-per-unique-word — typically a 5–10× reduction on real speech.

Two key micro-optimizations explained
Optimization 1 — Reduce memory overhead per candidate-path object Problem: The keyword search creates millions of partial candidate path objects per request. Each object normally carries a hidden internal dictionary for storing its attributes. That dictionary costs ~280 bytes per object — across millions of objects, this creates heavy memory pressure and slows down garbage collection significantly.
Fix: Declare all attributes up front so the runtime can store them in fixed-size slots instead of a dictionary. No dictionary is allocated at all. → Result: ~80 bytes per object (saves ~200 bytes each), allocation 30% faster
Optimization 2 — Eliminate repeated function dispatch in the inner loop Problem: The scoring function must choose between modes (e.g. "entropy-weighted" vs "flat"). This choice was made by comparing a mode string on every single call. That's 40+ string comparisons per call × millions of calls per request = compounding waste.
Fix: Decide which scoring variant to use once at the start of each request. Bind that specific variant as a ready-to-call function. The inner loop calls it directly — no mode checking, no string comparisons. → Result: zero dispatch overhead inside the inner loop

Async & Concurrent Inference

The server exposes three ways to call it: synchronous (wait for the result before doing anything else), async (send the request and let your application keep working while waiting), and parallel (fire multiple requests simultaneously via a worker pool). Choosing the right pattern determines whether you fully use the server's capacity or leave it under-utilized.

  • infer() Synchronous

    Blocking call — the application waits until the server returns the result. Correct for scripts, notebooks, and single-threaded batch jobs. Simple but cannot overlap with other work while waiting for the response.

  • infer_async() Asyncio

    Non-blocking coroutine for async application code. While waiting for the server response, the application is free to handle other tasks. No threads are created. Scales to many concurrent requests with minimal overhead.

  • infer_future() Thread Pool

    Dispatches the request to a background worker and returns a handle to the future result. Correct for synchronous code that needs to fire multiple requests at once. Up to 16 requests can be in-flight simultaneously via a shared worker pool.

Typical server latency vs. concurrency

The server's dynamic batcher absorbs burst traffic: requests arriving within the same 20 ms window are grouped into a single shared GPU pass. Sending 4–8 requests in parallel will often yield the same or better per-request latency as sequential calls — the server batches them at no extra cost. Beyond 16 concurrent requests the queue starts growing and worst-case latency rises.

Threading behaviour while waiting for the server

The network layer releases the application's threading lock while waiting for the server response. This means a background thread dispatched via the thread pool does not block the rest of the application during the server round-trip — only during the brief audio pre-processing (~0.5 ms) and result parsing (~0.2 ms). The async path behaves the same way — other tasks run freely while this one is waiting for its response.

Usage patterns — choosing the right calling style
Pattern 1  — Sequential  (~14 requests / second) For each audio clip in your list:     Send request → wait for result → process it → then move to the next Best for: scripts, notebooks, simple one-off tasks. Easy to reason about but does not use available concurrency.
Pattern 2  — Thread Pool  (~48 requests / second with 8 workers) Dispatch all requests to the worker pool at once The server may coalesce several into a single forward pass Collect results as they complete Best for: batch jobs, synchronous server code. Up to 16 requests can be in-flight simultaneously.
Pattern 3  — Async / Await  (~45 requests / second) Launch all requests as concurrent coroutines While awaiting responses, the event loop is free to handle other work All results collected once the last one completes Best for: async web servers (FastAPI, aiohttp). No threads created — scales to hundreds of concurrent requests.
Adding a timeout (thread pool pattern) Dispatch request → if no response within N seconds → raise a timeout error Catch the error and return a graceful fallback response instead Prevents a slow or unreachable server from hanging your application indefinitely.
Concurrency limit: The worker pool supports up to 16 simultaneous in-flight requests. If you dispatch more than 16 at once, the extras wait in the local queue — not on the server. For best throughput, keep concurrency at or below 16, aligned with the server's own batch capacity.

Putting It All Together

Each technique addresses a different failure mode. GPU compilation and dynamic batching give the fast decoder a 170× real-time factor. Batching strict-mode language-constrained requests together cut their latency by 47–49%. Overlapping CPU feature work with GPU decoding cut the full pipeline by 41%. Blank-token penalty and language filtering keep accuracy high at utterance boundaries. Keyword boosting recovers domain-specific vocabulary recall without any model retraining.

The unifying design principle is composability: every feature is independently togglable per request with sane defaults. A caller that needs none pays zero cost. A caller that needs all gets the full stack in a single API call with no model reload.

The path from 124 ms unoptimized to 72 ms fully-optimized balanced decoder — while simultaneously scaling maximum throughput 5× and halving stress latency — demonstrates that the highest-impact gains are rarely inside the model. They are in the scheduling, batching, and CPU/GPU overlap strategy wrapped around it.
